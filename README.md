# appchain-template

A template for running your own **Katana appchain**: a Starknet rollup sequenced by
[katana](https://github.com/dojoengine/katana), settling to Starknet (Sepolia or Mainnet)
through a [piltover](https://github.com/keep-starknet-strange/piltover) core contract,
with **TEE-attested settlement** â€” either a real **AMD SEV-SNP enclave** producing **SP1
Groth16 proofs** (Succinct network) verified on-chain, or a mock software attestation for
testing. Batteries included: systemd units, nginx TLS router, GitHub deploy workflows, and
a full Grafana/Prometheus/Loki monitoring stack with Slack alerting.

Everything is parameterized by **one file: [`appchain.conf`](appchain.conf)**.

## The deployment matrix

Two dimensions, four combos â€” each runs as its own systemd unit / state dir / port, all
coexisting on one host:

- **NETWORK** âˆˆ {`sepolia`, `mainnet`} â€” the Starknet settlement network
- **MODE** âˆˆ {`enclave`, `mock`} â€” `enclave`: katana inside an AMD SEV-SNP VM
  (`--tee sev-snp`, real SP1 proofs, real on-chain `AMDTeeRegistry`); `mock`: bare katana
  (`--tee mock`, software attestation, mock registry â€” no SEV-SNP hardware needed)

Unit names (`<chain-name>-<network>-<mode>`), RPC ports (`BASE_PORT`+n), and metrics ports
derive from `appchain.conf` â€” see [`scripts/lib/config.sh`](scripts/lib/config.sh).

The typical progression: **sepolia-mock** (cheap, no special hardware, real Sepolia
settlement) â†’ **sepolia-enclave** (real TEE + real proofs) â†’ **mainnet-enclave**
(production). Enclave internals are documented in [`TEE.md`](TEE.md).

## Quickstart

1. **Use this template** â†’ create your repo.
2. **Edit [`appchain.conf`](appchain.conf)** â€” chain name, chain ids, domains, host.
3. **Bootstrap the host** per [`HOST_SETUP.md`](HOST_SETUP.md) (mock mode: any Ubuntu box;
   enclave mode: an AMD SEV-SNP host) and point your DNS at it.
4. **Add GitHub secrets/vars** per [`SETUP.md`](SETUP.md) (settlement account, SSH key, â€¦).
5. **Actions â†’ Init Rollup** (one-time per combo; deploys piltover â€” real gas) â†’ review +
   merge the chain-config PR it opens.
6. **Actions â†’ Deploy** â†’ the node is up. Optionally run
   `scripts/deploy-monitoring.sh <user>@<host>` for Grafana/alerts.

The full step-by-step checklist (with secrets table and verification curls) is in
[`SETUP.md`](SETUP.md).

Using [Claude Code](https://claude.com/claude-code)? The repo ships a **`/setup-appchain`
skill** (`.claude/skills/setup-appchain/`) that drives this whole flow interactively â€”
open Claude Code in your new repo and ask it to "set up my appchain".

## Layout

| Path | What |
|---|---|
| `appchain.conf` | **the** config: chain identity, domains, ports, registries, version pins |
| `scripts/lib/config.sh` | derives the per-combo map (units/ports/registries) from appchain.conf |
| `scripts/init.sh` | one-time rollup bootstrap: deploys piltover, generates chain-config |
| `scripts/deploy.sh` | idempotent combo deploy: binary/image, unit, chain-config, nginx |
| `scripts/deploy-monitoring.sh` | deploys the monitoring stack + Grafana vhost |
| `scripts/appchainctl` | lifecycle CLI over SSH (`-n <network> -m <mode>` selects the combo) |
| `scripts/run-appchain.sh` / `run-enclave.sh` | the systemd-run launchers (mock / enclave) |
| `scripts/validate.sh` | offline validation of conf + templates (run by CI) |
| `host/*.tpl` | systemd unit + nginx vhost templates (rendered per combo at deploy) |
| `host/amdsev/` | SEV-SNP VM launcher (vendored from katana) |
| `host/monitoring/` | docker-compose stack: Prometheus, Loki, Grafana (+ alerts), exporters |
| `.github/workflows/` | `init.yml` (one-time, opens chain-config PR), `deploy.yml`, `validate.yml` |
| `chain-config-<net>-<mode>/` | generated by the Init workflow (committed via its PR) |
| `manifests/` | deploy/init receipts written back by the workflows |
| `HOST_SETUP.md` / `SETUP.md` / `TEE.md` | host bootstrap / consumer checklist / enclave internals |

## Operating

```sh
scripts/appchainctl status                    # sepolia-enclave (default combo)
scripts/appchainctl -n sepolia -m mock logs -f
scripts/appchainctl -n mainnet restart
NETWORK=mainnet MODE=enclave scripts/deploy.sh user@host   # redeploy a combo
RENDER_ONLY=1 scripts/deploy.sh                            # dry-run: render + plan, no ssh
```

Tunables (env on deploy, or workflow inputs): `BLOCK_TIME_MS` (block cadence),
`SETTLEMENT_BATCH_SIZE` (blocks per `update_state` â€” one SP1 proof covers the whole batch,
so bigger batches divide the proving + STRK cost per block; the price is settlement
latency), `IDLE_FLUSH_SECS` (max time between settlements on a slow chain).

## Caveat: the `init rollup` race

`katana init rollup` declares the piltover class and deploys it in the same run; the deploy
can lose the race to the declare's confirmation and revert with `Class â€¦ is not declared`.
The remedy is rerunning â€” `scripts/init.sh` retries automatically (`INIT_ATTEMPTS`,
default 3). A failed attempt deploys nothing, so retries can't leave duplicates.
